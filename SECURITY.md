# Security Policy

## Trust Boundaries

- Browser UI to local API.
- Local API to control plane.
- Control plane to OpenWrt adapter.
- OpenWrt adapter to data plane: dnsmasq, nftables, firewall4, Xray, Zapret.
- Secret files under `/etc/router-policy/secrets`.

## Default Exposure

The web/API listener defaults to localhost in development and must not be exposed to WAN. Any LAN binding must be explicit and protected by authentication, CSRF checks, and firewall rules.

There is no default administrator password. First setup uses a one-time setup token generated by `router-policy auth setup-token`; the token hash is stored with restricted permissions and is destroyed after administrator creation.

State-changing API requests require an `X-CSRF-Token` header that matches the HttpOnly session cookie.

## Secrets

The API must never return:

- VPN subscription URLs;
- VLESS UUIDs;
- REALITY private/public operational keys;
- Telegram bot tokens;
- passwords;
- full Xray configs containing credentials.

Secrets are displayed as masked values only.

Secret-bearing scripts and diagnostics must not use shell tracing (`set -x`),
dump complete Xray configurations, or include raw subscription payloads in
logs, crash bundles, API responses, or event streams. Public diagnostics must
be sanitized before publication.

## Local Audit

Run:

```sh
router-policy security audit
```

The audit is non-destructive and checks configuration gates, known unsafe defaults, and device-dependent unresolved items.

## Reporting

Do not include live subscription URLs, UUIDs, tokens, or full diagnostic archives in public reports.
